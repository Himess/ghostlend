## 0. Top 10 — architecture-critical for a confidential lending protocol

1. **There is no oracle push-callback anymore.** `FHE.requestDecryption` / `FHE.setDecryptionOracle` were deprecated in FHEVM v0.9 and are absent from all current docs. Public decryption is a **pull model**: `FHE.makePubliclyDecryptable(handle)` on-chain → off-chain `publicDecrypt(handles)` via relayer/SDK → **your own permissionless finalize function** that rebuilds the handle list from storage and calls `FHE.checkSignatures(handlesList, abiEncodedCleartexts, decryptionProof)`. Epoch aggregate publishing and liquidation booleans must be designed as *finalize transactions submitted by a keeper/bot/user*, not as callbacks the protocol waits to receive. (`SG /smart-contract/oracle.md`, `CL` v0.9)
2. **`FHE.checkSignatures` has NO replay protection** — the same `(handles, cleartexts, proof)` triple verifies twice. Every finalize function needs its own once-only state guard (epoch status enum, `finalized` flag). (`SG /smart-contract/functions.md`, `/smart-contract/oracle.md`)
3. **ERC-7984 transfers never revert on insufficient balance** — they silently move **encrypted 0** (all-or-nothing, no partial fill): `transferred = FHE.select(success, amount, 0)`. A pool MUST account with the **returned `transferred` handle** (or `after − before`), never the requested amount. (`OZ ERC7984.sol _update` + `FHESafeMath.tryDecrease`; `EX sealed-bid tutorial`)
4. **ACL is ephemeral by default**: FHE op results are usable by the contract only until function exit. `FHE.allowThis(...)` every stored handle, and `FHE.allow(handle, user)` **in addition** for anything the user must decrypt (dual grant). Forgetting either is the #1 cause of "works in one tx, dead in the next". (`SG /smart-contract/acl.md`, `/acl/acl_examples.md`)
5. **`require(FHE.isSenderAllowed(externalHandle))` on every user-supplied ciphertext** — otherwise an attacker passes the victim's balance handle as their own input and binary-searches its value from observable effects ("each successful or failed transfer leaks one bit"). (`SG /smart-contract/acl/acl_examples.md`)
6. **Token amounts are `euint64` with max 6 decimals** (wrapper caps at `_maxDecimals() = 6`; supply cap `type(uint64).max`). Interest-index math must fit euint64 (or hop to euint128 — mul cost explodes), and `FHE.div/rem` accept **plaintext divisors only**. Plaintext interest indexes multiplying encrypted principals is the right shape; encrypted-by-encrypted division is impossible. (`OZ ERC7984ERC20Wrapper.sol`; `SG /smart-contract/types.md`)
7. **HCU per-tx budgets bound your loops**: 20,000,000 global + 5,000,000 sequential-depth per transaction (documented for "devnet"; Sepolia numbers unpublished — HCULimit contract `0xa10998783c8CF88D886Bc30307e631D6686F0A22`). Non-scalar euint64 mul = 596k HCU, euint128 mul = 1.686M. Never iterate over all borrowers homomorphically in one tx. (`SG /development-guide/hcu.md`)
8. **Privacy leaks through behavior, not just values**: never revert on an encrypted condition (impossible anyway) and always execute **both legs** of a branch (one leg moving encrypted zero) — in lending: liquidation paths, collateral seizures, and repayments should perform the same external-call pattern regardless of the hidden outcome. (`SG /smart-contract/logics/loop.md`, `/logics/error_handling.md`)
9. **`FHE.makePubliclyDecryptable` is permanent and irrevocable** ("globally and permanently authorizing any entity"). Only ever mark *aggregates, booleans, or throwaway handles* — never a handle from which per-user balances can be derived. (`SG /smart-contract/oracle.md`)
10. **The proof binds handle ORDER**: proof for `[a,b]` fails for `[b,a]`; on-chain `handlesList` must be rebuilt **from contract storage** (never trusted from calldata) in the exact same order used off-chain, and each cleartext ABI-decodes as one 32-byte word per handle. (`SG /smart-contract/oracle.md`; `EX highest-die-roll` wrong-order + cross-game forgery tests)

## 1. Removed / deprecated APIs & version skew

- `FHE.requestDecryption`, `FHE.setDecryptionOracle`, request IDs, and the Gateway-pushed callback: **deprecated FHEVM v0.9, "must be removed"** — every older tutorial using them is stale. Even the archived v0.10–v0.12 doc trees were rewritten to the new model, so the old signatures are no longer documented anywhere. (`CL` v0.9; `?ask`)
- (History, for ports only) v0.8 changed the oracle callback signature to `callback(uint256 requestID, bytes cleartexts, bytes decryptionProof)` per ERC-7995 — if migrating pre-v0.9 code, expect that shape. (`CL` v0.8)
- **`SepoliaConfig` was removed in FHEVM v0.9.** Inherit `ZamaEthereumConfig` from `@fhevm/solidity/config/ZamaConfig.sol` — it auto-resolves protocol addresses via `block.chainid` for BOTH Ethereum mainnet and Sepolia; requires `@fhevm/solidity >= 0.9.1`. Don't mix old and new config bases. (`SG /smart-contract/configure.md`; `?ask` citing v0.9 migration)
- `InputVerifier` is NOT part of the inherited config; it is resolved at runtime via `FHEVMExecutor.getInputVerifierAddress()`. (`SG /smart-contract/configure.md`)
- `ebytesXX` types **removed** in v0.7; `einput` → `externalEuintXX`/`externalEbool`/`externalEaddress`; library renamed `TFHE` → `FHE`. (`CL` v0.7)
- v0.12 breaking set: new handle format (`FHE_comp` domain separator + prev blockhash + timestamp — never derive/hardcode handles), MultichainACL contracts deleted, `isUserDecryptionReady` signature changed, `ECDSA.sol` → `FhevmECDSA.sol` import rename, ACL error `ExpirationDateBeforeOneHour` → `ExpirationDateInThePast`. (`CL` v0.12)
- **Version skew:** Sepolia testnet runs FHEVM **v0.13**, Ethereum mainnet runs **v0.11** — v0.12/v0.13 features (HCU limit contract, `FHE.isPublicDecryptionResultValid`, all-contracts delegation sentinel `0xffff…ffff`) exist on Sepolia but NOT on mainnet. Protocol version numbers ≠ npm package versions (`@fhevm/solidity` npm latest is 0.11.1 — unrelated numbering). (`CL`)
- `FHE.sum` / `FHE.isIn` are announced in the v0.13 changelog but **absent from the Solidity API reference** (roadmap "ETA -") — do not design around them. (`CL` v0.13; `?ask` on functions.md)
- Overflow-checked encrypted integers: "coming soon" — not available. (`SG /smart-contract/types.md`)
- Relayer URL changed: current Sepolia relayer is `https://relayer.testnet.zama.org`; archived v0.10 docs show `relayer.testnet.zama.cloud` (dead — don't copy). (`SG /smart-contract/configure/contract_addresses.md` vs archived docs)
- **Package confusion (docs explicitly warn LLMs get this wrong):** `@zama-fhe/sdk` v3 (high-level `ZamaSDK` + React hooks, current default) ≠ legacy `@zama-fhe/relayer-sdk` (low-level `initSDK`/`createInstance`, still used by the Hardhat template & examples, npm-live at 0.4.4, NOT npm-deprecated). The changelog also names a future "`@fhevm/sdk`" refactor — **UNCERTAIN** whether that equals `@zama-fhe/sdk`; npm today ships `@zama-fhe/sdk@3.2.0`. Pin explicitly. (`SDK /migration/migrate-v2-to-v3.md`; `CL` v0.13)
- Since v0.12 `FHE.fromExternal` on an uninitialized/invalid handle returns **trivial-encrypt of 0 instead of reverting** — malformed input silently becomes zero; validate business-logic consequences (a "deposit 0" is harmless, a "repay 0" that clears a flag is not). (`CL` v0.12)

## 2. FHE types & arithmetic

- Arithmetic is **unchecked and wraps modulo 2^N** — no overflow error, by design (an error would leak information). Guard manually: compute `tmp`, `isOverflow = FHE.lt(tmp, old)` (for add), `FHE.select(isOverflow, old, tmp)`. (`SG /smart-contract/types.md`, `/smart-contract/operations.md`)
- `FHE.div` / `FHE.rem`: **plaintext divisor only**; encrypted rhs "will result in a panic". (`SG /smart-contract/types.md`)
- `euint256` supports **no arithmetic or ordering** (only bitwise, shifts/rotates, eq/ne, neg, not, select, rand). `eaddress` (= euint160) supports only eq/ne/select. Full arithmetic tops out at `euint128`. (`SG /smart-contract/types.md`)
- No signed encrypted integers exist (types table lists ebool/euint8..256/eaddress only). One `?ask` response mentioned "`eint*`" — treated as generation artifact. **UNCERTAIN only in that one response; assume nonexistent.** (`SG /smart-contract/types.md`; `_fragments/questions-fhe.md` #5)
- Shift/rotate amount is computed **modulo the bit width** of the left operand (`FHE.shr(euint64 x, 70)` shifts by 6) — differs from Solidity `>>`. (`SG /smart-contract/operations.md`)
- Mixed plaintext/encrypted operands: the plaintext side must not exceed the encrypted side's width (`add(uint8, euint8)` OK; `add(uint32, euint16)` invalid). (`SG /smart-contract/functions.md`)
- A comparison with a plaintext LEFT operand silently swaps operands and applies the **opposite** comparator (`gt(uint32 a, euint16 b)` computes `lt(b, a)`). Review generated logic carefully. (`SG /smart-contract/functions.md`)
- `FHE.neg` on unsigned types = **modular opposite**, not signed negation. (`SG /smart-contract/functions.md`)
- FHE operations are **state-changing**: no `view` functions, no `eth_call` computation of encrypted values; expose stored handles from views instead. (`SG /smart-contract/functions.md`)
- Uninitialized encrypted values compute as 0/false — use `FHE.isInitialized(v)` where the distinction matters (first deposit, first borrow). (`SG /smart-contract/functions.md`, `/smart-contract/configure.md`)
- Downcasting (`euint64 → euint32`) silently truncates high bits. (`SG /smart-contract/operations/casting.md`)
- **Trivial encryption (`FHE.asEuintXX(plain)`) is publicly readable on-chain** — "not secure in any sense"; use only for constants (zero handles, error codes). (`SG /smart-contract/operations/casting.md`, `/functions.md`)
- Cost hygiene: use the smallest type that fits; prefer scalar (plaintext) second operands over trivially-encrypted ones ("this will save a lot of gas"); `euint64` is the doc-recommended balance type. (`SG /smart-contract/operations.md`; `EX sealed-bid tutorial`)

## 3. HCU (FHE compute) limits

- Per-transaction limits (page says "current devnet"): **20,000,000 HCU total** and **5,000,000 HCU sequential depth**; exceeding either **reverts**. Split work across transactions or reduce dependency chains (depth binds long chains of dependent ops before the global cap). (`SG /development-guide/hcu.md`)
- **Sepolia-specific numbers are not documented.** Sepolia has `HCU_LIMIT_CONTRACT = 0xa10998783c8CF88D886Bc30307e631D6686F0A22` (v0.12+ adds per-block caps + whitelist) — read the deployed contract's getters to confirm real caps. (`SG contract_addresses.md`; `CL` v0.12; `?ask`)
- Cost cliffs to design around (full verbatim tables in `01-fhe-core.md`): non-scalar `mul` euint64 = 596,000; euint128 = 1,686,000; scalar `rem` euint64 = 1,153,000; euint128 = 1,943,000. A handful of wide mul/rem ops can consume the whole 5M depth budget — keep per-user updates O(few ops), never loop all borrowers. (`SG /development-guide/hcu.md`)
- Mock-side estimation exists: `hre.fhevm.computeTransactionHCU(receipt)` in `@fhevm/hardhat-plugin` — use it in tests to budget epochs/liquidations. (plugin types; `TPL`)

## 4. ACL rules & mistakes

- Results of FHE ops are **ephemerally** allowed to the computing contract and revoked at function exit — `FHE.allowThis(handle)` everything you store, every time it changes (each `FHE.select`/op produces a NEW handle needing a fresh grant). (`SG /smart-contract/acl.md`, `/logics/conditions.md`)
- **User decryption requires BOTH** `FHE.allowThis(handle)` AND `FHE.allow(handle, user)` — the docs ship a deliberately-wrong variant proving each alone fails ("dapp contract … is not authorized to user decrypt handle"). (`SG /acl/acl_examples.md`; `EX fhe-user-decrypt-single-value`)
- **Missing `FHE.isSenderAllowed(input)` = inference attack** (binary-searching someone else's handle through your contract's observable effects). Mandatory on every ciphertext argument callers pass in. (`SG /acl/acl_examples.md`)
- Encrypted constants created in a constructor need `FHE.allowThis` there, or they're unusable in later transactions. (`SG /logics/error_handling.md`)
- `FHE.allowTransient` = current-transaction only (EIP-1153) — right tool for handing a handle to a token/callee in-tx; wrong tool for anything read later. Before calling `token.confidentialTransferFrom(user, pool, amountHandle)` with the no-proof overload you must `FHE.allowTransient(amountHandle, address(token))`. (`SG /smart-contract/acl.md`, `/logics/loop.md`; `OZ` swap examples)
- Whether `FHE.allow` itself requires the caller to already hold permission on the handle is **not stated** in docs (it IS explicit for `makePubliclyDecryptable`). Safe design: only `allow` handles your contract produced or verified via `fromExternal` + `isSenderAllowed`. **UNCERTAIN.** (`?ask`; `_fragments/questions-fhe.md` #6)
- Denied accounts (`FHE.isAccountDenied`) can neither grant nor receive permissions. (`SG /smart-contract/acl.md`)
- **Delegation traps:** calling `FHE.delegateUserDecryption(delegate, address(this), exp)` from inside a contract "always reverts because `msg.sender == contractAddress`" — an EOA must call `IACL.delegateForUserDecryption` on the ACL directly. Invariants: delegator ≠ delegate ≠ contractAddress, expiry > now (and ≥ 1h ahead via SDK), max one delegate-or-revoke per block per (delegator, delegate, contract) tuple. On-chain delegation propagates L1 → Gateway in ~1–2 minutes; early delegated decrypts throw `DelegationNotPropagatedError`. (`SG /acl/delegation.md`; `SDK /guides/delegated-decryption.md`)
- **Reorg leak:** ACL events reach the Gateway on block inclusion — a `FHE.allow` in a reorged-out block can still leak. For high-value reveals use a two-step authorization with a **>95-block timelock** before the actual `allow`. (`SG /acl/reorgs_handling.md`)

## 5. Handles

- Only guarantee: equal handles → equal plaintexts (and different plaintexts → different handles). NOT guaranteed: different handles → different plaintexts; equal plaintexts → equal handles. Never compare/dedupe logic via handle bytes — use `FHE.eq`. (`SG /smart-contract/handles.md`)
- Same computation in different blocks yields **different handles** (prev blockhash mixed into derivation, v0.12+). (`SG /smart-contract/handles.md`; `CL` v0.12)
- **Ciphertext-uniqueness assumption:** OZ's wrapper keys unwrap requests by the burned-amount handle with an explicit code warning ("this holds here but is not always true. Be cautious when assuming cipher-text uniqueness") — same caveat applies to any pool mapping keyed by returned handles. (`OZ ERC7984ERC20Wrapper.sol _unwrap`; `PA confidential-wrapper.md`)

## 6. Branching, control flow & behavioral privacy

- Encrypted conditions **cannot revert** ("Transactions do not revert if a condition fails"). Neutralize failures with `FHE.select` and surface them via the encrypted last-error idiom (+ event with an error index). Verbatim pattern in `01-fhe-core.md` §7. (`SG /logics/error_handling.md`)
- No `break` on encrypted conditions — constant-bound loops with `FHE.select` inside. Encrypted-index array access costs a full O(n) homomorphic scan — "should be avoided whenever possible". (`SG /logics/loop.md`)
- Transitioning encrypted → public logic is **always async** (makePubliclyDecryptable → off-chain publicDecrypt → checkSignatures); there is no synchronous decrypt inside a contract. Design state machines around it. (`SG /logics/conditions.md`)
- **Do both legs every time**: in AMM/lending flows run the transfer on both directions (one encrypted-zero) or observers learn the branch taken from which external calls happened. (`SG /logics/loop.md`)

## 7. Public decryption & finalize-function trust

- Flow: (1) contract has ACL on handle (typically `allowThis`) → `FHE.makePubliclyDecryptable(handle)` (no request id, no fee, **permanent + irrevocable**); (2) off-chain `publicDecrypt(handles)` → `{ clearValues, abiEncodedClearValues, decryptionProof }`; (3) permissionless finalize tx → rebuild `bytes32[] handlesList` via `FHE.toBytes32` **from storage** → `FHE.checkSignatures(handlesList, abiEncodedCleartexts, decryptionProof)`. No `FHE.allow(oracle)` of any kind is needed. (`SG /smart-contract/oracle.md`; `?ask`)
- `checkSignatures` reverts on: empty/short proof, signatures below KMS threshold, unregistered signer (`KMSInvalidSigner(address)`); success emits `PublicDecryptionVerified`. View twin: `FHE.isPublicDecryptionResultValid(...)` (v0.12+ → Sepolia yes, mainnet v0.11 **no**). (`SG /smart-contract/functions.md`; `CL`)
- **No replay protection** — add your own finalize-once guard. **Anyone can call** the finalize function — spoofed cleartexts revert (proof binds cleartexts to handles), but submission can be frontrun/griefed: never attach caller privileges or caller-dependent effects to finalize. (`SG functions.md`; `?ask`; `EX heads-or-tails`)
- Proof is bound to handle **order**; cross-request proofs verify only against their own handles → rebuilding `handlesList` from storage prevents an attacker binding valid-but-wrong values to your state (cross-game forgery test demonstrates). Cleartexts ABI-encode as **one 32-byte word per handle** (two euint8 == `abi.encode(uint256,uint256)`) — decode types must match per-word. (`EX highest-die-roll`, `heads-or-tails`)
- Docs typo trap: oracle.md tutorial reads `results.values[handle]`; the real `PublicDecryptResults` field is **`clearValues`**. (`SG oracle.md` vs `EX heads-or-tails`)
- `makePubliclyDecryptable` return type: oracle.md says `internal` (void), functions.md says `internal returns (T)` — examples ignore any return. **UNCERTAIN**; verify against `@fhevm/solidity` source. (`_fragments/questions-decryption.md` #6)
- **No latency SLA anywhere** for Sepolia public decryption (two `?ask` attempts failed); the only adjacent figure is "2–5 second FHE decryption" for user-decrypt balance reads. Design async-no-deadline; measure empirically. (`?ask`; `SDK /guides/check-balances.md`)
- No documented max handles per `publicDecrypt`; relayer throttling surfaces as HTTP 429 → `RelayerRequestFailedError` (`.retryable`, `.retryAfter` — the latter often CORS-hidden in browsers; implement own backoff). Keep epoch batches small and chunkable. (`SDK /api-references/sdk/errors.md`; `?ask`)
- **Failure recovery is free**: nothing is pending on-chain; `makePubliclyDecryptable` never expires; re-call `publicDecrypt` anytime; only flip state after `checkSignatures`. (`?ask`)
- **Liveness dependency**: relayer/KMS downtime = no decryption at all (public or user) until service returns — liquidation/epoch flows must tolerate stalls (grace periods, no hard deadlines tied to decryption). (`SDK /concepts/security-model.md`)
- `checkSignatures` gas is undocumented ("optimized via signature caching" — semantics unexplained); measure finalize-tx gas for multi-value epochs. (`SG functions.md`; `_fragments/questions-decryption.md` #5, #12)

## 8. User decryption, permits & the browser runtime

- Dual ACL grant required (§4). Event-logged handles are NOT decryptable by neutral indexers without on-chain rights or delegation (`delegatedDecryptValues`). (`SDK alpha /guides/decrypt-from-event-logs.md` — ALPHA channel)
- **Permit model (SDK v3):** additive EIP-712 signatures, **max 10 contracts per permit signature**, default TTL 30 days; transport key pair TTL 30 days (plaintext in IndexedDB — XSS/extension risk); wallet disconnect or account switch wipes permits, chain switch keeps them; expiry silently reprompts. Batch all your protocol's contract addresses into ONE `grantPermit` call. (`SDK /concepts/permit-model.md`, `/concepts/security-model.md`)
- Never fire decrypt hooks on render: `useDecryptValues` is **disabled by default** (`enabled: true` required) and prompts an EIP-712 wallet signature if no permit is cached ("unexpected MetaMask popup… potential Blockaid flags"); gate on `useHasPermit` + explicit user action; permits must cover EVERY contract in the query or it stays disabled. (`SDK /api-references/react/usedecryptvalues.md`, `/guides/encrypt-decrypt.md`)
- Browser runtime: COOP/COEP headers for SharedArrayBuffer multithreading (silent single-thread fallback otherwise; 4–8 threads sweet spot), CSP `worker-src blob:` + `script-src 'wasm-unsafe-eval'` + `connect-src https://cdn.zama.org`, client-side only (SSR crashes), keep WASM integrity check on; FHE artifacts are multi-MB (IndexedDB-cached in browser; **in-memory only** for `node()` — pass custom storage or re-download on restart). (`SDK /concepts/security-model.md`, `/api-references/sdk/relayerweb.md`, `/api-references/sdk/fheartifactcache.md`)

## 9. ERC-7984 token semantics (OZ v0.5.1)

- **Clamping everywhere:** insufficient-balance transfer → encrypted 0 moved, tx succeeds (`_update` + `FHESafeMath.tryDecrease`); mint overflowing total supply → mints 0 (`tryIncrease`); unwrap from insufficient balance → burns 0 and finalizes to a 0 payout. README verbatim: "many functions do not revert on failure as they would in normal contracts." (`OZ ERC7984.sol`, `FHESafeMath.sol`, `README.md`)
- **Deployed-vs-library divergence:** OZ v0.5.0 removed the revert for never-funded senders, but Zama's *deployed* ConfidentialWrapper docs (governing cUSDCMock/cWETHMock) still document `ERC7984ZeroBalance(holder)` for fresh accounts as `from` (transfers AND unwrap). The deployed wrappers are **UUPS-upgradeable proxies** — behavior can change. Tolerate BOTH: never assume a fresh-account pull reverts or succeeds. **UNCERTAIN which library version the Sepolia mocks run.** (`OZ CHANGELOG.md` 0.5.0; `PA confidential-tokens/confidential-wrapper.md`)
- **Operator = unlimited amount, time-bounded only**: "Setting an operator … allows the operator to *take all of your tokens*." Expiry is `uint48` unix-seconds, **inclusive** (`block.timestamp <= until`); `isOperator(holder, holder)` always true; SDK `setOperator` defaults to **1 hour** — pools needing longer windows must pass explicit `until`; `OperatorSet` is a public event (reveals the user→pool authorization graph + timing). (`OZ ERC7984.sol`, `docs token.adoc`; `SDK /guides/operator-approvals.md`)
- SDK guide claims transfer approval ≠ unshield approval ("does not automatically allow them to unshield") but on-chain both paths check the same `isOperator` — **UNCERTAIN** (product-flow guidance vs deployed-wrapper difference); test unwrap-as-operator on Sepolia if the pool must unshield for users. (`SDK operator-approvals.md` §6 vs `OZ ERC7984ERC20Wrapper.sol`)
- Operators cannot decrypt other users' balances — an operator "can only know success after a transaction (by decrypting the transferred amount)". No pre-checks; design pull-then-verify. (`OZ docs token.adoc`)
- **ACL on the returned `transferred` handle:** persistent `FHE.allow` for `from`, `to`, and the token; the *operator caller* gets only `allowTransient` (same-tx use). Pool-as-`to` can use it in-tx, but every derived stored handle (updated collateral/debt) needs fresh `FHE.allowThis` (+ `FHE.allow(.., user)`). (`OZ ERC7984.sol _update/_transfer`; `?ask ask-acl-returned-handle`)
- The euint64 no-proof overloads require caller ACL on the amount handle AND the token to be allowed on it (`FHE.allowTransient(amount, token)` first) — else `ERC7984UnauthorizedUseOfEncryptedAmount`/ACL failure. Input proofs bind to (contract, user): encrypt against the **token** address when calling the token's `externalEuint64` overload directly, or against the **pool** address when the pool does `FHE.fromExternal` itself. Wrong binding = proof verification failure. (`OZ` swap examples; `EX erc7984.md` tests)
- **Transfer-and-call hazards:** balances update BEFORE the receiver hook runs (reentrancy surface — repo ships `ERC7984ReentrantMock`); refund on `false` is **best-effort** (a receiver can drain its balance in the hook and still return false — "the sender's tokens end up with the recipient rather than being refunded"); never manually refund AND return false (double refund); the receiver must grant the token read access on its returned `ebool` (`FHE.allowTransient(result, msg.sender)`); contracts not implementing `IERC7984Receiver` revert the `AndCall` variants but **plain transfers to them succeed and can strand funds** (no code check). (`OZ ERC7984.sol`, `IERC7984Receiver.sol`, `ERC7984Utils.sol`)
- Privacy-by-design leaks: public `from`/`to` on every `ConfidentialTransfer`, `OperatorSet` params, `Wrap.roundedAmount` cleartext, `UnwrapFinalized.cleartextAmount`, `AmountDisclosed.amount`, and `AndCall` emitting send+refund as two events. (`OZ IERC7984*.sol`; `SDK event-decoders`)
- Library stability: docs index says "not formally audited and not covered by the bug bounty program" while README says npm installs "the latest audited release" and `audits/` holds six per-release PDFs (through `2026-06-v0.5.pdf`). No backward-compat guarantees; ERC-7984 itself is a **DRAFT** ERC. Peer dep is an **exact pin** `@fhevm/solidity: 0.11.1`. Pin `@openzeppelin/confidential-contracts@0.5.1`. (`OZ README/docs/audits`; eips.ethereum.org)

## 10. Wrapper & registry

- Wrapper decimals = `min(underlying.decimals(), 6)`; `rate = 10^(underlyingDecimals − 6)`; 18-dec underlyings get rate 10^12 → sub-10^-6-token amounts unrepresentable. Base `ERC7984.decimals()` is hard-coded 6 (cUSDC maps 1:1; cWETH divides by 10^12 in UI). (`OZ ERC7984ERC20Wrapper.sol`, `ERC7984.sol`; `PA confidential-wrapper.md`)
- **Wrap rounds DOWN** to the nearest `rate()` multiple; excess refunded (ERC-1363 path) or not pulled (`wrap()` path); "If the amount is less than the rate, the wrapping will succeed but the recipient will receive 0 confidential tokens" — dust danger. Wrap reverts `ERC7984TotalSupplyOverflow` near the uint64 cap (and that check "may revert even if the confidentialTotalSupply did not overflow" — inflatable by direct underlying transfers to the wrapper). (`OZ wrap()/onTransferReceived()`; `PA confidential-wrapper.md`)
- **Unwrap is two-step async**: `unwrap` burns immediately, pays NOTHING; underlying moves only on `finalizeUnwrap(requestId, cleartext, decryptionProof)` after public decryption — funds sit in limbo between steps (SDK `resumeUnshield` exists); `finalizeUnwrap` **publicly reveals the cleartext amount** (event) — unshield privacy is pre-finalization only. Request id IS the burned-amount ciphertext handle (uniqueness caveat §5). (`OZ ERC7984ERC20Wrapper.sol`; `PA confidential-wrapper.md`)
- `wrap()`'s returned handle carries only `allowTransient` for `msg.sender` — same-tx use only. (`OZ`)
- `inferredTotalSupply()` can be inflated by direct underlying transfers and lags during unwrap-limbo — not a solvency oracle. (`OZ`)
- **Non-standard underlyings unsupported**: fee-on-transfer/deflationary (undercollateralization), rebasing up (yield accrues to wrapper — "Shielded … staking shares do not earn rewards") or down (undercollateralization), multi-entry-point (double-wrap), ERC-777 hooks (reentrancy), **pausable/blocklist tokens (USDC!) can freeze wrap/unwrap**, flash-mintable distorts inferred supply. (`OZ` natspec; `PA confidential-wrapper.md` table)
- Registry: entries can be **revoked permanently** ("the token can never be re-registered") — "Always verify the `isValid` flag", not just a non-zero wrapper address. Registry + wrappers are DAO-owned UUPS proxies. (`PA wrapper-registry.md`, `confidential-wrapper.md`)

## Lending-design implications (derived from the above — each traces to sourced bullets)

- **Deposits/repayments:** user `setOperator(pool, until)` (explicit `until`; SDK default is 1h) → user submits encrypted amount + proof bound to the right (contract,user) pair → pool `confidentialTransferFrom` → **account only the returned `transferred` handle** (clamping) → `FHE.allowThis` + `FHE.allow(user)` on every updated position handle.
- **Interest:** plaintext indexes (RAY-style) multiplying encrypted principals; division by plaintext only; mind euint64 range (6 decimals) and HCU mul costs; avoid encrypted×encrypted mul in hot paths.
- **Liquidation/epochs:** mark only aggregate/boolean handles publicly decryptable; keeper submits finalize tx with proof; finalize = permissionless + replay-guarded + order-stable handle list from storage; both-legs execution so outcomes don't leak via call patterns; no hard deadlines on decryption (liveness).
- **Never**: revert on encrypted conditions, loop all borrowers homomorphically, trust caller-supplied handle lists in finalize, expose raw user balance handles to `makePubliclyDecryptable`, or assume the deployed wrappers behave exactly like OZ v0.5.1.
