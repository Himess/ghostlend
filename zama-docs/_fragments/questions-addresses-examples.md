# Unresolved questions — addresses + examples pass (2026-07-03)

Questions that remain open after the page dumps and `?ask=` queries.

1. **Mainnet relayer URL**: the `?ask` response confirms the Zama-hosted mainnet relayer requires an API key, but no mainnet relayer URL appears on any fetched page. Where is it published / how is the key obtained? (Likely answered in https://docs.zama.org/protocol/sdk/guides/relayer-api-keys.md — not dumped in this pass.)
2. **Mainnet host-contract gaps**: solidity-guides `contract_addresses.md` lists only ACL, FHEVM_EXECUTOR, and KMS_VERIFIER for Ethereum mainnet — no InputVerifier, DecryptionOracle (DECRYPTION_ADDRESS), HCU_LIMIT, INPUT_VERIFICATION_ADDRESS, or GATEWAY_CHAIN_ID equivalents. Are those simply not deployed on mainnet v0.11, or published elsewhere (SDK network presets)?
3. **Per-token decimals for the Sepolia mocks**: neither the Sepolia table nor the `?ask` answers state the decimals of the underlying USDC/WETH mocks or their wrappers (rules cap wrappers at 6). Confirm on-chain via `wrapper.decimals()` / `wrapper.rate()` / underlying `decimals()` before hardcoding amounts (e.g., is the WETH mock 18-decimals with rate 10^12, per the ZAMA/cZAMA example row?).
4. **Mint limit semantics**: docs state the public `mint` is "limited to 1,000,000 tokens per call" — is that 1,000,000 whole tokens (scaled by decimals) or 1e6 base units, and is there any cooldown/per-address cap? Not specified.
5. **Sepolia gas ETH**: Zama docs cover token minting + shielding but say nothing about obtaining Sepolia ETH for gas (standard third-party faucets assumed). No Zama-run ETH faucet found.
6. **Which SDK package for a new project**: examples/tests import `@zama-fhe/relayer-sdk`, but the v0.13 changelog introduces `@fhevm/sdk` as its refactored replacement. Which one does the current `fhevm-hardhat-template` pin today, and is `@zama-fhe/relayer-sdk` deprecated?
7. **DecryptionOracle naming**: `DECRYPTION_ADDRESS` (0x5D8BD78e2ea6bbE41f26dFe9fdaEAa349e077478) is confirmed by `?ask` as "the DecryptionOracle address" on Sepolia, but it lives on the Gateway-facing decryption path — whether a contract ever needs to reference it directly under the current (post-v0.9, relayer-based) flow is unclear; current examples never use it (they use `FHE.makePubliclyDecryptable` + `FHE.checkSignatures`, addresses auto-resolved by `ZamaEthereumConfig`).
8. **HCU numeric limits**: v0.12 introduced configurable per-block / per-tx / per-depth HCU caps and a whitelist, but no concrete numeric limits for Sepolia are published in the fetched pages (the HCU_LIMIT contract address is known; query it on-chain or find the solidity-guides HCU page).
9. **ERC7984Mock availability**: the VestingWallet test deploys `ERC7984Mock` and the wrapper test deploys `ERC20Mock` — these mock contracts are referenced but their source is not included in the example pages (they come from the OZ confidential-contracts repo / template); confirm exact import paths when reproducing tests.
10. **cUSDCMock operator/permit surface**: examples show `setOperator(address, uint48Timestamp)` on ERC7984 tokens; whether the deployed Sepolia cUSDCMock exposes any additional disclosure functions (`requestDiscloseEncryptedAmount` etc. from the ConfidentialWrapper spec) — the deployed mocks follow the protocol-apps ConfidentialWrapper (UUPS), not the simpler OZ example contract; verify ABI on Etherscan before integrating.
