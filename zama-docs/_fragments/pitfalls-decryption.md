# Pitfalls — decryption (FHEVM / Zama SDK), one bullet each + source

- **The oracle-callback API is gone.** `FHE.requestDecryption` + `FHE.setDecryptionOracle` + request id + automatic callback were deprecated in FHEVM v0.9 and do not exist in the current flow; you must run your own off-chain finalizer that calls `publicDecrypt` and submits results to a self-defined function. — ?ask q3 (cites https://docs.zama.org/protocol/changelog/zama-protocol-change-log.md ); https://docs.zama.org/protocol/solidity-guides/smart-contract/oracle.md
- **`FHE.makePubliclyDecryptable` is global, permanent and irrevocable** — "globally and permanently authorizing any entity"; never mark a handle you might want private later. — https://docs.zama.org/protocol/solidity-guides/smart-contract/oracle.md
- **ACL prerequisite:** the calling contract must already have ACL permission on the handle before `makePubliclyDecryptable` (typically via `FHE.allowThis`); no `FHE.allow(oracleAddress)` is needed for anything. — https://docs.zama.org/protocol/solidity-guides/smart-contract/oracle.md ; ?ask q6
- **`FHE.checkSignatures` gives NO replay protection** — the same `(handles, cleartexts, proof)` triple can be submitted twice; your finalize function must implement a finalize-once/state guard. — https://docs.zama.org/protocol/solidity-guides/smart-contract/functions.md (hint block); oracle.md warning
- **Anyone can call your finalize function** — design it permissionless-safe; spoofed cleartexts revert (`KMSInvalidSigner(address)`) because the proof binds cleartexts to the exact handles, but griefing/frontrunning of the *submission* is possible, so don't attach caller privileges to it. — ?ask q2; https://docs.zama.org/protocol/examples/basic/decryption/heads-or-tails.md
- **Ordering is cryptographic:** the decryption proof is bound to the handle order passed to `publicDecrypt`; `handlesList` and the ABI-encoded cleartexts must use the identical order or verification reverts (`KMSInvalidSigner`). — https://docs.zama.org/protocol/solidity-guides/smart-contract/oracle.md ; highest-die-roll.md wrong-order test
- **Rebuild `handlesList` from contract storage, not calldata** — proofs from another request verify fine against *their own* handles, so trusting caller-supplied handles lets an attacker bind the wrong values to your state; both doc examples read handles from `games[gameId]` storage. — https://docs.zama.org/protocol/examples/basic/decryption/heads-or-tails.md (cross-game forgery test)
- **Cleartext ABI widths:** the relayer ABI-encodes each clear value as one 32-byte word in handle order (two euint8s == `encode(["uint256","uint256"])`); on-chain `abi.encode`/`abi.decode` types must match per-word or bytes differ / decode reverts. — https://docs.zama.org/protocol/examples/basic/decryption/highest-die-roll.md
- **Docs typo trap:** oracle.md tutorial reads `results.values[handle]` but the real `PublicDecryptResults` property is `clearValues`. — https://docs.zama.org/protocol/solidity-guides/smart-contract/oracle.md vs heads-or-tails.md
- **`checkSignatures` revert set:** empty/invalid-length proof, valid signatures below KMS threshold, or any non-registered KMS signer; success emits `PublicDecryptionVerified(handlesList, abiEncodedCleartexts)`. — https://docs.zama.org/protocol/solidity-guides/smart-contract/functions.md
- **No latency SLA anywhere:** no documented Sepolia latency or block-count expectation for public decryption; only figure in docs is "the 2-5 second FHE decryption" for user-decrypt balance reads — measure empirically, design for unbounded async. — ?ask q1, q8; https://docs.zama.org/protocol/sdk/guides/check-balances.md
- **No documented max handles per `publicDecrypt`** and relayer rate-limits surface as HTTP 429 → `RelayerRequestFailedError` (`.retryable`, `.retryAfter` — the latter often unavailable in browsers due to CORS; implement your own backoff). — ?ask q4; https://docs.zama.org/protocol/sdk/api-references/sdk/errors.md
- **No retry/refund mechanism exists or is needed:** nothing is pending on-chain; `makePubliclyDecryptable` never expires; just call `publicDecrypt` again after failures and only flip state after `checkSignatures` succeeds. — ?ask q5
- **Relayer/KMS downtime = no decryption at all** (public or user); on-chain data stays safe but unreadable until service returns — a liveness dependency your liquidation flow must tolerate. — https://docs.zama.org/protocol/sdk/concepts/security-model.md
- **User decryption needs BOTH ACL grants:** `FHE.allowThis(handle)` AND `FHE.allow(handle, user)`; forgetting `allowThis` fails with "dapp contract ... is not authorized to user decrypt handle ...". — https://docs.zama.org/protocol/examples/basic/decryption/fhe-user-decrypt-single-value.md
- **FHE ops (incl. ACL/makePubliclyDecryptable) are state-changing** — cannot run in `view` functions / `eth_call`; return handles from view functions instead. — https://docs.zama.org/protocol/solidity-guides/smart-contract/functions.md
- **Never fire decrypt hooks on render:** `useDecryptValues`/`useConfidentialBalance` prompt an EIP-712 wallet signature if no permit is cached ("unexpected MetaMask popup, user rejection, potential Blockaid flags"); gate on `useHasPermit` + explicit button. — https://docs.zama.org/protocol/sdk/guides/encrypt-decrypt.md
- **Permits chunk at 10 contracts per EIP-712 signature** — batching more addresses means multiple wallet prompts; batch all expected contracts into one `grantPermit` call. — https://docs.zama.org/protocol/sdk/concepts/permit-model.md
- **TTLs:** transport key pair default 30 days (`transportKeyPairTTL`, seconds) and permits default 30 days (`permitTTL`, days); expiry silently requires a wallet re-sign; wallet disconnect/account-switch wipes permits (chain switch keeps them). — https://docs.zama.org/protocol/sdk/concepts/permit-model.md ; usedecryptvalues.md
- **Transport private key is stored in plaintext** (IndexedDB) — same-origin XSS, physical access, or malicious extensions can read it. — https://docs.zama.org/protocol/sdk/concepts/security-model.md
- **Delegation propagation lag:** on-chain delegation must sync L1 → Gateway (Arbitrum), typically 1-2 minutes; early delegated decrypts throw `DelegationNotPropagatedError`; expiration must be ≥1 h ahead (`DelegationExpirationTooSoonError`); constraints `delegator≠delegate≠contractAddress` and one delegate-or-revoke per block per tuple; EOAs must call the ACL directly (`IACL.delegateForUserDecryption`), not `FHE.delegateUserDecryption`. — https://docs.zama.org/protocol/sdk/guides/delegated-decryption.md ; https://docs.zama.org/protocol/solidity-guides/smart-contract/acl/delegation.md
- **Publicly-logged handle ≠ publicly decryptable:** event-log encrypted values still require the signer to hold on-chain decrypt rights (or delegation); neutral indexers fail without `delegatedDecryptValues`. — https://docs.zama.org/protocol/sdk/alpha/guides/decrypt-from-event-logs.md (ALPHA)
- **Package confusion:** `@zama-fhe/sdk` v3 (high-level, `ZamaSDK`/hooks) ≠ legacy `@zama-fhe/relayer-sdk` (low-level `createInstance`/`initSDK`); docs explicitly warn AI agents mix them up; hardhat plugin still re-exports relayer-sdk types. — https://docs.zama.org/protocol/sdk/migration/migrate-v2-to-v3.md
- **Mock-mode boundary:** example decryption suites hard-fail outside mock (`hre.fhevm.isMock`); mock `publicDecrypt` proofs are accepted by `checkSignatures` only in the mock env — real Sepolia runs need the real relayer; there is no `awaitDecryptionOracle` helper. — heads-or-tails.md; ?ask q7
- **Browser runtime requirements for SDK decrypt/encrypt:** COOP/COEP headers for `SharedArrayBuffer` (else slow single-thread fallback), CSP `worker-src blob:` + `script-src 'wasm-unsafe-eval'` + `connect-src https://cdn.zama.org`, client-side only (SSR breaks), WASM integrity check should stay enabled. — https://docs.zama.org/protocol/sdk/concepts/security-model.md ; encrypt-decrypt.md
- **`useDecryptValues` requires permits covering EVERY input contract or the query stays disabled/prompts** — grant all addresses up front. — https://docs.zama.org/protocol/sdk/api-references/react/usedecryptvalues.md ; usegrantpermit.md
