# Unresolved questions (after ?ask= attempts) — FHE core docs, 2026-07-03

1. **Exact HCU limits on Ethereum Sepolia.** hcu.md only states "The current devnet has an HCU limit of **20,000,000** per transaction and an HCU depth limit of **5,000,000** per transaction." A ?ask query answered: "The docs **do not provide Sepolia-specific HCU limit values**… I cannot confirm from the docs that Sepolia matches those same values." Sepolia does have `HCU_LIMIT_CONTRACT` = 0xa10998783c8CF88D886Bc30307e631D6686F0A22 — the deployed contract's getters could be read on-chain to confirm. Also unknown: the v0.12 per-block HCU cap value (`hcuCapPerBlock`) on Sepolia.

2. **Which `@fhevm/solidity` npm version the current unversioned docs describe.** Docs pin no version (?ask: "The Solidity guides in this docs set do not specify an npm version"). npm latest = 0.11.1 (2026-02-19), while the protocol changelog says Testnet runs FHEVM v0.13 and Mainnet FHEVM v0.11 (different numbering scheme). Working assumption: install `@fhevm/solidity@0.11.1` (latest) + the fhevm-hardhat-template; but there is no doc statement mapping package 0.11.1 ↔ protocol v0.13/v0.12 features (e.g. whether `FHE.isPublicDecryptionResultValid`, listed in the current API reference and introduced in protocol v0.12, is in package 0.11.1).

3. **Availability of `FHE.sum` / `FHE.isIn`.** Changelog lists them as FHEVM v0.13 features "(Copro)"; current Testnet = v0.13. But ?ask on functions.md: "I cannot find `FHE.sum` or `FHE.isIn` as available, supported operators in the current Solidity operations documented" and the Roadmap lists `FHE.isIn()` with "ETA -". No Solidity signatures exist anywhere in the docs. Treat as unavailable.

4. **Input-proof packing ceilings in the CURRENT protocol.** Current inputs.md documents packing but no numeric limits (?ask confirmed "the limits are not documented"). Only ARCHIVED v0.10 coprocessor docs give defaults via ?ask: `maximum-handles-per-input` = 255 and CRS `max-num-bits` "usually 2048". Whether Sepolia's deployment today uses exactly 255/2048 is unconfirmed.

5. **Signed encrypted integers (`eintX`).** One ?ask response parenthetically claimed the current types include "`eint*` (signed integers)", but the authoritative types.md table lists only ebool/euint8-256/euint160(eaddress) and no signed types appear in the HCU tables or API reference. **UNCERTAIN — assume signed types do NOT exist**; the ?ask mention looks like an answer-generation artifact.

6. **Whether `FHE.allow` itself requires the caller to already hold ACL permission on the handle.** ?ask: "For `FHE.allow(...)` specifically, the docs don't state a general 'caller must already have ACL permission on that handle' rule." (It IS explicit for `makePubliclyDecryptable`.) Given the `isSenderAllowed` inference-attack guidance, safest design: only ever `allow` handles the contract itself produced or verified via `fromExternal`/`isSenderAllowed`.

7. **`ebool` "Bit Length 2".** types.md's table says Ebool bit length is 2 while functions.md says the backend represents booleans as an 8-bit encrypted uint. Cosmetic inconsistency; no practical impact identified, but exact ebool encoding is unclear from docs.

8. **Sepolia relayer/SDK operational limits** (rate limits of `https://relayer.testnet.zama.org`, publicDecrypt latency, userDecrypt quotas) are not covered in the solidity-guides section at all — they live in the SDK/relayer docs, which were out of scope for this pass.
