### P8 — HCU limits (Sepolia)

**Documented / source caps** (from installed `@fhevm/host-contracts@0.10.0`, whose `HCULimit.sol` is internally **v0.1.0**):
- `MAX_HOMOMORPHIC_COMPUTE_UNITS_PER_TX = 20,000,000` (global per-tx)
- `MAX_HOMOMORPHIC_COMPUTE_UNITS_DEPTH_PER_TX = 5,000,000` (sequential depth per-tx)
- No per-block cap in the v0.1.0 source. Both are `private constant`.

**⚠️ Version gap (new finding):** the **deployed** Sepolia HCULimit proxy `0xa10998783c8CF88D886Bc30307e631D6686F0A22` (impl `0xcc7b81e598fcf5e2247f29de87c6c879d06581e2`) reports **`HCULimit v0.3.0`** — a *newer* contract than the v0.1.0 source shipped in the latest npm package (0.10.0). Its cap constants are `private` and **not exposed by any getter** (6 candidate getters — `MAX_HOMOMORPHIC_COMPUTE_UNITS_PER_TX`, `maxHomomorphicComputeUnitsPerTransaction`, `...PerBlock`, etc. — all revert). So the exact **deployed** caps cannot be read on-chain and are **not confirmed** to equal 20M/5M; the v0.12+ changelog notes a per-block HCU cap was added in a later version, which v0.3.0 may enforce.

**What IS confirmed — real measured HCU on live Sepolia** (`fhevm.computeTransactionHCU` works on real receipts, not just mock):
- deposit `pull` tx (fromExternal + allowTransient + confidentialTransferFrom + FHE.add + allowThis + allow): **globalHCU = 748,096 ; maxHCUDepth = 369,000**.
- This is ~3.7% of the 20M global figure and ~7.4% of the 5M depth figure → **ample headroom**; a per-user lending op (deposit/borrow/repay with a handful of FHE ops) is nowhere near any plausible cap. The risk is only in *unbounded loops* (e.g. iterating all borrowers), which the design already avoids.

**KMS detail:** the P6 public-decrypt proof carried **numSigners = 7** (proof layout `1 + 65·numSigners + extraData` ⇒ ≥ 456 bytes) — the Sepolia KMS returns 7 signatures.

**Net (pack UNCERTAIN #1):** partially resolved. The *documented* per-tx caps are 20M global / 5M depth; the *deployed* v0.3.0 caps are unreadable and may add a per-block cap. Practically, measured deposit cost leaves a large margin, so size lending ops by keeping FHE-op count per tx small and never looping over all positions.
